---
title: "Pairing and connecting to Flic 2 buttons"
description: "Step-by-step guide to scanning for Flic 2 buttons, completing cryptographic pairing, and establishing verified BLE sessions for your app."
---

Connecting a Flic 2 button to your app involves two distinct flows: a **full verify** that runs once when you pair a new button, and a **quick verify** that runs on every subsequent reconnection. Both flows establish a shared session key that secures all further communication. The full verify performs a cryptographic handshake using Ed25519 signature verification and X25519 key exchange; the quick verify reuses the stored pairing key with a lightweight Chaskey-LTS cipher. Understanding both flows is essential before implementing the protocol layer.

## Prerequisites

Before scanning for a button, it must be in **Public mode** so it advertises the pairing service identifier your scanner looks for.

- **New or factory-reset button**: press and hold for at least 1.5 seconds.
- **Button with existing pairings**: press and hold for at least 6 seconds.

Public mode lasts up to 30 seconds. If the button reverts to Private mode before pairing completes, repeat the hold gesture.

<Info>
  In Public mode, the button accepts full verify requests from any app. In Private mode, only apps that already hold a valid pairing key can connect.
</Info>

## Full verify flow

Full verify runs the first time your app connects to a button. It authenticates the button using a known Ed25519 public key and establishes a session key through an X25519 key exchange.

<Steps>
  <Step title="Scan for the button">
    Perform a BLE scan and filter for advertisements containing the Flic 2 pairing service UUID. When you find a match, record the device address. The button must be in Public mode for the pairing service to appear in its advertisements.
  </Step>

  <Step title="Send FullVerifyRequest1">
    Open a BLE connection to the button and send a `FullVerifyRequest1` packet. Include a `tmp_id` — a random value you generate to identify this request within the session.
  </Step>

  <Step title="Validate FullVerifyResponse1">
    The button responds with a `FullVerifyResponse1` containing:

    - An **Ed25519 signature** over the response payload.
    - An **ECDH public key** for the X25519 exchange.
    - **Random bytes** generated by the button.
    - The button's **BLE address**.

    Verify the Ed25519 signature using the following public key:

    ```text
    d33f2440dd54b31b2e1dcf40132efa41d8f8a7474168df4008f5a95fb3b0d022
    ```

    Also confirm that the BLE address in the response matches the device address you scanned. Abort and set `STATE_FAILED` if either check fails.
  </Step>

  <Step title="Perform X25519 key exchange and derive session keys">
    Generate a client X25519 key pair. Compute the shared secret using your private key and the button's ECDH public key from step 3.

    Derive `fullVerifySecret` by computing SHA-256 over the shared secret concatenated with a `0x00` byte (or `0x80` for Flic Duo — see [Flic Duo pairing](/guides/flic-duo#pairing-with-flic-duo-support)).

    From `fullVerifySecret`, derive two values using HMAC-SHA-256:

    | Value | HMAC message |
    |---|---|
    | `verifier` | `'AT'` |
    | `sessionKey` | `'SK'` |
  </Step>

  <Step title="Send FullVerifyRequest2">
    Send a `FullVerifyRequest2` packet containing:

    - Your **client ECDH public key**.
    - **Client random bytes** (fresh random bytes you generate for this request).
    - The **verifier** derived in the previous step.

    The button uses the verifier to confirm your app computed the same shared secret.
  </Step>

  <Step title="Handle FullVerifyResponse2">
    On success, the button responds with `FullVerifyResponse2` containing:

    - A **button UUID** — stable identifier for this physical button.
    - The button's **display name**.
    - Current **firmware version**.
    - Current **battery level**.
    - The button's **serial number**.

    On failure, you receive a `FullVerifyFailResponse` instead. See [Error handling](#error-handling) below.
  </Step>

  <Step title="Store pairing credentials">
    Derive the pairing id and pairing key from `fullVerifySecret` using `fullVerifyHmac` with the message `'PK'`. Store both values persistently — you need them for every future quick verify.
  </Step>

  <Step title="Session established">
    The session is now in `STATE_SESSION_ESTABLISHED`. The `sessionKey` derived in step 4 is active for this session. You can now initialize button events and send commands.
  </Step>
</Steps>

## Quick verify flow

Quick verify runs every time a previously paired button reconnects. It is much faster than full verify because both sides already share a pairing key.

<Steps>
  <Step title="Send QuickVerifyRequest">
    Send a `QuickVerifyRequest` packet containing the `pairing_identifier` you stored during full verify. If you support Flic Duo, also set `supports_duo=1` in the request (this changes the session key derivation — see [Flic Duo](/guides/flic-duo)).
  </Step>

  <Step title="Receive QuickVerifyResponse">
    The button responds with a `QuickVerifyResponse` containing its own random bytes and, if `supports_duo=1` was set, an `is_duo` flag.
  </Step>

  <Step title="Derive session key">
    Combine your client random bytes with the button's random bytes, then compute the session key using **Chaskey-LTS** keyed with your stored pairing key. The result is the session key for this connection.
  </Step>

  <Step title="Session established">
    The connection advances to `STATE_SESSION_ESTABLISHED`. Proceed to initialize button events.
  </Step>
</Steps>

## Error handling

### FullVerifyFailResponse reasons

If full verify fails, the button sends a `FullVerifyFailResponse`. The two most common reasons are:

- **`INVALID_VERIFIER`** — the verifier in your `FullVerifyRequest2` did not match what the button computed. This usually means a key derivation error on the client side.
- **`NOT_IN_PUBLIC_MODE`** — the button left Public mode before the handshake completed. Prompt the user to re-enter Public mode and retry.

### No logical connection slot available

If the button sends a `NoLogicalConnectionSlotsInd`, it has no capacity to accept another logical connection on this BLE connection. Your app can open at most **two concurrent requests per logical connection id**. Release an existing slot before retrying.

### QuickVerifyNegativeResponse

If quick verify fails, the button sends a `QuickVerifyNegativeResponse`. This may mean the button no longer recognizes your pairing key — for example, after a factory reset.

**Test whether the button is truly unpaired:**

1. Attempt full verify to check if the button is in Public mode.
2. If the button is not in Public mode, the user must factory-reset or re-enter Public mode manually.
3. If full verify succeeds, store the new pairing credentials and discard the old ones.

<Warning>
  Do not silently delete stored pairing credentials on a `QuickVerifyNegativeResponse` without first confirming through full verify that the button has actually been reset. The failure could be transient.
</Warning>

## Connection state machine

The verify flow progresses through the following states:

| State | Description |
|---|---|
| `STATE_WAIT_FULL_VERIFY1` | Waiting for `FullVerifyResponse1` after sending `FullVerifyRequest1`. |
| `STATE_WAIT_FULL_VERIFY2` | Waiting for `FullVerifyResponse2` after sending `FullVerifyRequest2`. |
| `STATE_WAIT_QUICK_VERIFY` | Waiting for `QuickVerifyResponse` after sending `QuickVerifyRequest`. |
| `STATE_SESSION_ESTABLISHED` | Session is active; button events and commands are available. |
| `STATE_FAILED` | An unrecoverable error occurred. Close the connection. |

<Note>
  A single BLE connection can host multiple logical connections, each with its own verify flow. Each logical connection id supports at most **two concurrent in-flight requests**. If your app and another app share the same BLE connection to a button (multi-app support), each counts against this limit independently.
</Note>
